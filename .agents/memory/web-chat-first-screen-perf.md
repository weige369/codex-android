---
name: Web-chat first-screen perf measurement
description: How LAN-phone first-screen load cost is measured and where the baseline lives.
---

First-screen (connection overlay) download cost is measured by `web-chat/scripts/measure-first-screen.mjs` (npm: `measure`, `measure:baseline`, `measure:check`). It is a dependency-free static analyzer of `web-chat/dist`, NOT a headless browser — no puppeteer/playwright is installed.

**What counts as first-screen:** whatever Vite marks eager in `dist/index.html` — the entry `<script type=module>` plus every `<link rel=modulepreload>` chunk, plus the stylesheet. Everything else in `dist/assets/*.js` is treated as deferred (lazy). Markdown, AgentActivityPanel, CodexVersionManager are intentionally lazy (see ChatScreenContent.tsx + MarkdownRenderer.tsx) and must stay OFF the eager list, or the markdown vendor bundle (~43 kB gzip) re-enters the critical path.

**Rule:** the saved baseline (`web-chat/perf/first-screen-baseline.json` + `first-screen-report.md`) reflects a specific build. After an *intentional* first-screen change, rebuild then `npm run measure:baseline` to refresh it. `measure:check` fails (exit 1) if first-screen gzip JS grows >5% vs baseline.

**Why:** the static script needs no browser, so exact byte sizes (raw/gzip/brotli) + a transparent network-time estimate are the always-available signal; the byte numbers are exact, the time figures are a network-only TTI lower bound (parse/exec excluded).

**Keep heavy effects off the eager tree:** `liquid-glass-react` (the `glass` manualChunk, ~32 kB gzip) was on the first-screen critical path only because `GlassSurface` (reached from the eager AIChatScreen→ChatScreenContent→ChatArea tree) imported it statically. Fix: split the LiquidGlass-dependent rendering into `GlassSurfaceLiquid.tsx` (default export) and have `GlassSurface` `lazy()`-import it, rendering a plain-div fallback when there's no glass variant. Result: first-screen JS gzip 137→105 kB (-24%), Slow-3G TTI ~12.5s→11.4s. **Rule:** any node_modules dep that gets its own manualChunk must only be reachable via dynamic import, or it re-enters the overlay critical path.

**Real-browser variant:** `web-chat/scripts/measure-first-screen-browser.mjs` (npm: `measure:browser`, `measure:browser:baseline`) drives headless Chromium (puppeteer-core, a **root** devDependency — if `node_modules/puppeteer-core` is missing run `npm install` at the repo root, NOT in web-chat; chromium is a nix system dep, auto-detected or via `CHROMIUM_BIN`) with phone viewport + CPU + network throttling, recording real FP/FCP/overlay-ready/TTI/TBT and cross-checking against the static baseline's network estimate.

**TTI / "ready to use" definition (don't regress to plain quiet-window-after-FCP):** the heavy parse/exec long tasks finish *just before* FCP on throttled profiles, so a quiet-window search anchored strictly after FCP finds nothing and TTI collapses to FCP. Fix (in `computeTti`): anchor at `max(FCP, overlayReady)` (overlayReady = `#web-chat-token` present, always > FCP) then walk long tasks forward extending through any starting before a 2s quiet window — so TTI = when the control exists AND the thread settled, and still catches trailing tasks. Plus `computeTbt` reports Total Blocking Time = Σ(longtask.dur−50) over **all** observed tasks **including pre-FCP** — that's the only metric that surfaces the parse/exec busyness FCP/TTI hide. Both medianed over samples; baseline+report carry `totalBlockingTimeMs`. It serves `dist` over a throwaway localhost HTTP server and navigates with `waitUntil:'domcontentloaded'` (the window `load` event never fires with no backend — pending requests). Heavy deps are isolated (dynamic import + graceful failure) so the static `measure` still runs without a browser. Baseline/report: `first-screen-browser-baseline.json` / `first-screen-browser-report.md`. Measured reality (May 2026): parse/exec+RTT adds ~+0.4s on LAN to ~+11s on Slow-3G/low-end over the network-only estimate — i.e. the static estimate badly understates phone TTI.
